Organizing Hooks – שמירה על סדר

למה חשוב לארגן Hooks?

ארגון נכון מונע בלגן ומשפר קריאות, תחזוקה ושימוש חוזר.

  • קריאות: איתור מהיר של ה־Hook המתאים.
  • תחזוקה: תיקונים ממוקדים בלי לשבור אזורים אחרים.
  • שימוש חוזר: קל לייבא ולהטמיע בפיצ׳רים נוספים.

כמה שיטות לארגון Hooks

תיקיית Hooks מרכזית בכל פיצ׳ר

בפיצ׳ר users לדוגמה (עם מיקום יחסי לשאר התיקיות):

src/
├─ features/
│  ├─ users/
│  │  ├─ components/
│  │  ├─ hooks/
│  │  │  ├─ useUsers.ts
│  │  │  ├─ useUserForm.ts
│  │  │  └─ index.ts
│  │  ├─ services/
│  │  └─ index.ts
│  └─ ...
└─ shared/ ...

קובץ index.ts יכול לייצא את כל ה־Hooks כך שהייבוא נשאר נקי: 

export * from './useUsers';
export * from './useUserForm';

ייבוא לדוגמה:

// מתוך קובץ בפיצ׳ר users
import { useUsers } from './hooks';

// או ממחוץ לפיצ׳ר (כאשר יש barrel בשורש הפיצ׳ר)
import { useUsers } from './features/users';

Shared Hooks

Hooks כלל־מערכתיים (שאינם תלויי פיצ׳ר) ממוקמים תחת shared/hooks: 

src/
├─ shared/
│  └─ hooks/
│     ├─ useFetch.ts
│     ├─ useDebounce.ts
│     └─ index.ts
└─ features/
   └─ users/ ...

ייבוא לדוגמה:

// מכל מקום באפליקציה
import { useDebounce } from '../shared/hooks';

// ואם יש barrel לשורש shared
import { useDebounce } from './shared';

כך ניתן להשתמש בהם בכל פיצ׳ר בלי להכפיל קוד.

שמות Hooks ברורים

  • תמיד prefix use — לדוגמה useAuth, useUsers.
  • השם מתאר בדיוק את הפעולה — כדי שמיד ברור מתי ואיך להשתמש.

Keep it small and modular

  • כל Hook מממש פונקציונליות אחת.
  • Hook גדל מדי? מפצלים ל־Hooks קטנים וברורים יותר.

סיכום

ארגון מסודר ל־Hooks שומר על קוד נקי, קריא ומודולרי. יחד עם Feature-Based Structure ו־Barrel Exports מתקבלת מערכת מסודרת שמוכנה להתרחבות.
