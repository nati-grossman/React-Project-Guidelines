Context API – מתי כן ומתי לא

מה זה Context API?

מנגנון לשיתוף נתונים ופונקציות בין קומפוננטות בלי להעביר props דרך כל העץ. יוצרים "קונטקסט" ומספקים אותו לכל רכיב שצריך גישה אליו.

מתי כן להשתמש

  • State גלובלי קטן: נתונים שנדרשים בכמה מקומות, כמו מצב התחברות (user/token), שפה (locale), או Theme (Dark/Light).
  • הימנעות מ־Props Drilling: כשמידע צריך לעבור הרבה רמות בעץ.
  • שילוב עם Component Composition: רכיבי UI מקבלים גישה לנתונים גלובליים בלי תלות בהיררכיה שלהם.

מתי לא להשתמש

  • עדכונים תכופים: state שמשתנה כל הזמן (timer, mouse position, scroll) עלול לגרום לרינדורים רבים אצל כל הצורכים.
  • נתונים ספציפיים לפיצ׳ר: השאירו מקומי ב־useState או ב־Custom Hook.
  • ניהול State מורכב: Context לא מחליף Redux/Zustand כאשר יש לוגיקה עסקית רבה או סנכרון נתונים מורכב.

טיפים לשימוש נכון

  • הפרידו לוגיקה מ־UI: עטפו לוגיקה ב־Custom Hook והשתמשו בו בתוך ה־Provider.
  • פצלו Contextים: צרו קונטקסטים נפרדים לפי סוג מידע (AuthContext, ThemeContext) כדי לצמצם רינדורים.

דוגמה קצרה: Provider ו־Consumer

// auth-context.tsx
const AuthContext = React.createContext<{ user: User | null }>({ user: null });

export function AuthProvider({ children }: { children: React.ReactNode }) {
  const { user } = useAuthLogic(); // Custom Hook שמטפל בלוגיקה
  return <AuthContext.Provider value={{ user }}>{children}</AuthContext.Provider>;
}

export function useAuth() {
  return React.useContext(AuthContext);
}
// Component.tsx
function HeaderUser() {
  const { user } = useAuth();
  return <span>{user ? user.name : 'Guest'}</span>;
}

סיכום

  • כן: מידע גלובלי ויציב, שימוש בכמה מקומות.
  • לא: עדכונים תכופים או נתונים מקומיים לפיצ׳ר.
  • כלי משלים: Context אינו תחליף מלא לניהול State מורכב.
